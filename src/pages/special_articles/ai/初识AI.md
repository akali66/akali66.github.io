# 初识AI

## 第一部分：从模型到API

AI为什么能被调用？目前绝大多开发者都不需要去造AI，而是去使用AI，而云端 API 服务便是最主流、最快捷的方式

这是绝大多数公司和开发者起步的选择。AI 巨头将训练好的强大模型放在云端，开放出简单的 HTTP 接口供你调用。你按使用量付费。

除此之外，你可以选择开源模型 + 本地/自有服务器部署（更自主、更可控）：如果你对数据隐私要求极高，或者有特殊的定制化需求，可以选择这条路。你需要自己准备算力（GPU服务器）和运维知识。

典型框架和模型：
框架：PyTorch, TensorFlow, Transformers (Hugging Face)。
模型库：Hugging Face Hub 上有成千上万的免费开源模型，涵盖 NLP、CV、语音等所有领域。

## 第二部分：概念工具

这部分先了解一下LLM、LangChain、MCP的概念

抽象一点，想象你要建一个智能厨房：

- LLM（大型语言模型）：一个极其聪明、但有点不接地气的“厨师长”。他熟读世界上所有的菜谱，能和你讨论任何美食理论。但你问他“我家冰箱里只有鸡蛋和番茄，能做什么？”，他可能会回答“番茄炒蛋”，也可能天马行空地说“分子料理番茄蛋羹”（因为他知识太泛了，不知道你家的实际情况）。

- LangChain：一个万能的“厨房总管”或“工作流设计图”。他知道厨师长（LLM）的优缺点。当你想点菜时，总管会做一系列安排：
    1. 先打开冰箱（工具调用），看看有什么食材（查询数据库）。
    2. 查一下你的饮食禁忌（检索记忆）。
    3. 把“现有食材”和“你的忌口”写成一张明确的清单（组装提示词），交给厨师长。
    4. 厨师长根据这张具体清单，给出靠谱的“番茄炒蛋”菜谱。
    5. 总管拿到菜谱后，可能还会用计算器算一下营养（调用计算工具），然后才把最终结果交给你。
    
    LangChain的核心价值是：把调用LLM的复杂流程标准化、自动化，让它能按步骤使用工具、访问数据。

- MCP（模型上下文协议）：一套统一的“工具插口标准”。在LangChain的厨房里，冰箱、计算器、食谱库可能牌子都不一样，连接方式五花八门。MCP就像规定了所有工具都必须用“USB-C接口”。这样，任何符合MCP标准的工具（无论谁生产的），都能即插即用地接入任何支持MCP的系统（比如LangChain未来的版本，或者其他AI框架）。它目标是解决工具集成的碎片化问题。

接下来再分开讲：

### LLM：大型语言模型

**它是什么？**

一个通过海量文本训练出来的、拥有数百亿甚至万亿参数的概率预测模型。你可以把它理解为一个“超级文本补全工具”。

- 输入：一段文本（叫做“提示词”或Prompt）。
- 输出：根据它从训练数据中学到的模式和规律，生成接下来最可能出现的文本。

关键点：

- 没有真正的“理解”：它只是在做数学计算，预测下一个词的概率。但因为数据量极大，效果上显得像“理解”了。
- 知识有“保质期”：它的知识截止于训练数据的时间。不知道之后的事（除非后续微调或通过外部信息输入）。
- 会“幻觉”：会一本正经地胡说八道，生成看似合理但实际错误的内容。因为它追求的是“像人一样流畅”，而不是“事实正确”。
- 上下文窗口有限：一次性能处理的文本长度有限制（比如4K、8K、128K Token）。超过就记不住了。

但当你问一个复杂问题时，问题就来了：

```javascript
// 假设你想问：“我公司Q3的销售额相比Q2增长了多少百分比？”
const badAnswer = await askLLM("我公司Q3的销售额相比Q2增长了多少百分比？");
// LLM会懵，因为它：1. 不知道“你公司”是谁。2. 没有销售数据。
// 它很可能开始幻觉，编造一个数字。
```

为了解决这个问题，我们需要让LLM能访问外部工具和数据。这就引出了 LangChain。

### LangChain：编排LLM的“指挥”

解决单纯调用LLM API能力不足的问题。LLM本身：

1. 无法获取实时信息（比如最新天气、股价）。
2. 无法进行精确计算、逻辑推理。
3. 无法操作外部系统（发邮件、查数据库）。
4. 无法处理长文本（比如一本你上传的PDF手册）。

**LangChain 的核心思想**：不把LLM当作一个“问答机”，而是当作一个决策大脑。由这个大脑来决定何时、以及如何使用其他工具来完成一个复杂任务。

它的核心概念：

- 链：把多个步骤组合起来。最简单的链是 LLMChain，就是“提示词 + LLM调用”。复杂的链可以包含多个LLM调用、工具调用、数据处理的步骤。
- 代理：这是LangChain的灵魂。代理 = LLM + 工具 + 决策逻辑。
    - 你给代理一个目标（比如“查一下北京今天的天气，然后用中文写首诗赞美它”）。
    - 代理内部的LLM会自己思考：“要完成这个目标，我需要分几步？先用哪个工具？”
    - 然后它自主选择调用“天气查询工具”，拿到数据后，再选择调用“写诗”的LLM能力，最终生成结果。
- 工具：给LLM扩展能力的函数。比如：
    - 搜索引擎工具：获取实时信息。
    - 计算器工具：做精确数学计算。
    - 数据库查询工具：执行SQL。
    - 自定义工具：任何你能用代码写的功能。
- 记忆：让LLM在多次对话中记住上下文。可以是简单的对话历史，也可以是更复杂的向量数据库存储。

LangChain 是一个功能强大的框架，帮你组装基于LLM的智能体。​ 但它也有个问题：工具生态比较碎片化，每个人定义的工具接口不一样，难以复用。于是，MCP 试图来解决这个问题。
