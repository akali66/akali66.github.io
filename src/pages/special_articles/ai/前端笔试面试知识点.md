## 第一部分 html

首先我认为，笔试面试的核心在你对框架的认识，html可能仅仅只会占一小部分，或者只是提一嘴，所以我准备大致讲一些核心或常见知识点：

### html的本质

你可以理解为一棵“语义树”（DOM树），每个标签（元素）是树上的一个节点，是有特定含义的

那`<div>`和`<span>`就是属于“无意义”的通用节点

### 语义化

在HTML5之前，写代码就是用多个`<div>`来布局，如`<div id="header">`、`<div class="nav">`，对机器（搜索引擎、屏幕阅读器）来说，其实都是一堆没有任何意义的div

而语义化标签的引入，就让结构本身有了意义，等于每个标签的“人设”

常用的语义化标签有（可以先简单过一下）：

- `<header>`：页面或章节的页眉。通常包含网站Logo、主导航、搜索框等。注意，一个页面可以有多个 `<header>`，比如文章内也可以有一个小的 `<header>`包含文章标题和元信息。
- `<nav>`：主导航区域。包含一组重要的链接，用于跳转到站内其他页面（如顶部导航、侧边栏导航）。不应将所有链接组都放入 `<nav>`，只用于主要导航区块。
- `<main>`：页面的核心、独立内容。每个页面有且仅有一个 `<main>`。其内容应是该页面独有的，不应包含全局性、重复出现的内容（如导航、页脚）。
- `<article>`：独立、完整、可独立分发的内容块。比如一篇博客文章、一条论坛帖子、一则新闻、一个独立的微件。关键在于，把它单独拿出来放到另一个上下文中，它仍然是完整、有意义的。
- `<section>`：文档中的一个主题性分组。通常带有一个标题。它用于将相关内容组织在一起。与 `<article>`的区别是，`<section>`的内容不一定能独立存在。比如，一本书的每一章可以用 `<section>`，而书中的每一篇文章则用 `<article>`。
- `<aside>`：与主要内容间接相关的内容。比如侧边栏、批注、广告、相关文章列表等。它应独立于其周围的内容。
- `<footer>`：页面或章节的页脚。通常包含版权信息、联系方式、相关文档链接等。

**为什么推荐用语义化标签？**
1. 对开发者友好：简单，易读，有规范。很好理解
2. 对SEO友好：搜索引擎的爬虫会分析页面结构，看到（`<main>`,`<article>`,`<h1>`）能帮爬虫更准确的理解内容和权重关系
3. 对无障碍访问友好：屏幕阅读器用户依赖这些标签来理解页面结构

### 元信息

讲完整体的，其中在构建html的时候就会遇到`<head>`标签里存在的元信息，它们是不显示但及其重要的，但是目前用的不多，所以大致提几个：

- `<meta charset=“UTF-8”>`：指定文档字符编码，避免乱码。必须放在 `<head>`的最前面
- `<meta name=“viewport” content=“width=device-width, initial-scale=1.0”>`：移动端适配的基石。它告诉浏览器视口宽度等于设备宽度，并且初始不缩放。


### 可访问性(a11y)

可访问性不是可选项，而是构建现代Web应用的责任。HTML 是 a11y 的第一道防线。(官方的话，不是我的话)

比较常见的就是
`alt`：`<img>`的替代文本。不仅是SEO，更是为视障用户描述图片内容。如果图片是装饰性的，可以写 alt=“”，但绝不能省略。

**ARIA 属性**：当 HTML 语义不足以描述交互时，用 ARIA 来补充。
- `role`：定义元素的角色，如 role=“button”、role=“dialog”、role=“navigation”
- `aria-label`：为元素提供一个不可见的标签。当元素本身没有文本内容时特别有用。

```javascript
<button aria-label="关闭弹窗" onclick="closeModal()">X</button>
<!-- 屏幕阅读器会读出“关闭弹窗 按钮”，而不是“X 按钮” -->
```

- `aria-labelledby`：通过ID关联到另一个元素，将其作为自己的标签

### 其他

这些大致就是html需要提出来讲的点，那复习完这几个点后，可能在面试中还会问到你一些别的内容：

1. `<!DOCTYPE html>`的作用是什么？
- 它不是标签，是为了声明。告诉浏览器我这个网页是遵循HYML5规范的。写了这个，就会触发浏览器的**标准模式**，而不是**怪异模式**，所以它必须放在HTML文档的第一行

2. 行内元素和块级元素的根本区别是什么？
- 首先他们两个在布局方式、内容模型上都有不同：
- **块级元素**独占一行，从上到下垂直排列，宽度默认填满父元素，可设置宽度高度内外边距。如`<div>`,`<p>`,`<h1>`,`<ul>`,`<li>`
- **行内元素**不独占一行，排在同一行直到排不下再换行，其宽度、高度、上下外边距、上下内边距设置无效。宽度和高度由其内容决定，如`<span>`、`<a>`、`<strong>`、`<em>`、`<img>`、`<input>`
- 内容模型中，块级元素通常可以包含其他块级元素和行内元素。行内元素通常只能包含数据或其他行内元素（例如，`<a>`里不能再放一个 `<a>`）。
- display属性：可以通过CSS的 display属性改变其默认行为（如 display: inline-block;使其兼具两者特性）。

3. src和 href的区别？
这其实是一个关于资源加载行为的问题
- `src(Source)`：表示“引入资源到当前文档中”，当解析到带src（如 `<script>`、`<img>`、`<iframe>`）的元素，则**会暂停其他资源的下载和处理**，直到该资源加载、解析、执行完毕。理解为“嵌入”关系
- `href(Hypertext Reference)`:表示“当前文档与外部资源之间建立链接”。浏览器在解析到带有 href的元素（如 `<link>`、`<a>`）时，会并行下载该资源，不会停止对当前文档的处理。它是一种“关联”关系。

4. 前端性能优化中，HTML 层面能做什么？
这个是一种分析题，首先你得从想象浏览器的加载，会经过哪些过程，如果不知道，后面会讲到，这里暂时过一下：
- 首先浏览器加载会渲染DOM树，那么最直接的优化方法就是尽可能的减少DOM树
那么渲染时就会涉及到html，css和js的加载，那同样可以根据这个点来思考一下三点：
- 多使用语义化标签，有助于浏览器更高效解析
- 将CSS放在头部，可以避免页面渲染后再应用样式，导致样式闪烁
- js放在底部（`<body>`之前），或使用async/defer属性，来避免js阻塞html解析和页面渲染
相对你可以相到的一个更好的点是：
- 利用 preload/prefetch：`<link rel=“preload”>`用于提前加载当前页面必定会用到的关键资源（如首屏字体、关键CSS/JS）。`<link rel=“prefetch”>`用于提前加载下一个页面可能用到的资源。

## 第二部分 JS

一般的流程是讲css，但是我认为，这个时候有必要先把js拿出来讲，因为个人感觉css大致都是一些样式上的知识点，这不都是对着设计图敲嘛？所以暂时先放后

那最基本的点就是ES6引入的内容了：

### 变量声明：var, let, const 到底有什么区别？

1. `var`最古老的，问题最多,第一想到的就是函数作用域和变量提升
- **作用域**：函数作用域，在整个函数内有效
```javascript
function testVar() {
    if (true) {
        var a = 10; // 在if这个块里声明的
    }
    console.log(a); // 输出 10,因为var是函数作用域，在函数内都能访问到。
}
testVar();
```
- **变量提升**：`var`声明的变量会被“提升”到其所在作用域的顶部。注意，提升的是声明，不是赋值。
```javascript
console.log(myVar); // 输出 undefined，而不是报错
var myVar = 5;
// 实际执行顺序被JS引擎理解为：
// var myVar; // 声明被提升到最前面
// console.log(myVar); // 此时myVar是undefined
// myVar = 5; // 赋值留在原地
```
- 可以重复声明：`var a = 1; var a = 2;`不会报错，这容易导致代码错误。

2. `let`和`const`(ES6引入，现代写法)
相对比之下对应的是：块级作用域、变量提升？（暂时性死区），不可重复声明
- **块级作用域**：块级作用域。{}大括号内就是一个块，如 `if`、`for`、`while`或单独的大括号。
```javascript
if (true) {
    let b = 20;
    const c = 30;
}
console.log(b); // 报错：b is not defined。因为b只在if这个块里有效。
console.log(c); // 报错：c is not defined。
```
- **变量提升？**：他们其实也有提升，但是和var不一样的是，声明之前会报错，叫做“暂时性死区”
```javascript
console.log(myLet); // 报错：Cannot access 'myLet' before initialization
let myLet = 5;
```
- 不可重复声明：在同一作用域内，`let a = 1; let a = 2;`会报语法错误。

对于`let`vs`const`:
- `let`用于声明值可变的变量。
- `const`用于声明常量。**它保证的是变量指向的内存地址不变**，而不是值不变。
    - 对于基本类型（数字、字符串、布尔等），值就保存在那个地址里，所以 `const PI = 3.14; PI = 3.15;`会报错。
    - 对于引用类型（对象、数组），地址不变，但对象内部的内容可以改变。
```javascript
const person = { name: '小明' };
person.name = '小红'; // 这是允许的,修改对象内部属性。
console.log(person); // { name: '小红' }

person = { name: '小刚' }; // 报错,试图给person赋一个新的对象（新地址）。
```

那其实总结下来，你要知道的点就是：首先，var是函数作用域，且是有变量提升，可以重复声明的，是历史遗留的问题。而let和const是块级作用域，有暂时性死区，不可重复声明。而const用于声明常量，保证其引用不变（再深入解释一下）,而现在基本上默认都是使用const，只有当你知道这个变量需要被重新赋值的时候，就会用let，基本不用var

### 闭包

js的重点，也是笔试面试的重点，很多都会考到

那想了解闭包，就先了解一下作用域：作用域是变量和函数可以被访问的范围，可以理解为，一个房子，你在你自己房间（局部作用域）放的东西,在客厅（全局作用域）的人不一定能直接拿到

那闭包就是：**当一个函数可以记住并访问其所在的词法作用域，即使这个函数在其词法作用域之外执行，就产生了闭包。**

简单来说：函数“包”住了外面的变量

举个例子：
```javascript
function createCounter() {
    let count = 0; // count 是 createCounter 函数的局部变量
    return function() {
        count++;   // 内部函数访问了外部函数的变量count
        console.log(count);
    };
}
const counter = createCounter(); // createCounter执行完，按说count该被销毁了
counter(); // 输出 1
counter(); // 输出 2
counter(); // 输出 3
```

内部函数一直保持这对`count`的引用，垃圾回收机制不会回收它，这个`count`变量就变成了一个私有变量，只能通过`counter`函数来修改

那这么看闭包会使没有用还扫不到的垃圾越来越多，那它有什么用呢？

闭包是函数和其周围状态（词法环境）的引用捆绑在一起。它的主要用途是创建私有变量和函数，常用于模块化、防抖节流等场景。

闭包的应用：节流、防抖（后面再讲）、私有化

### this

重要考点之一，四大绑定规则
- 默认绑定：独立函数调用。在非严格模式下，`this`指向全局对象（浏览器中是window）；严格模式下是`undefined`。
```javascript
function foo() { console.log(this); }
foo(); // 浏览器中输出 undefined (严格模式)
```
- 隐式绑定：函数作为对象的方法被调用。this指向调用它的那个对象。
```javascript
const obj = {
    name: ‘前端’,
    sayName: function() { console.log(this.name); }
};
obj.sayName(); // 输出 ‘前端’，this指向obj
// 坑：丢失绑定
const fn = obj.sayName;
fn(); // 输出 undefined，因为现在是独立调用，this指向全局（或undefined）
```
- 显式绑定：用 `call`, `apply`, `bind`强行指定`this`
```javascript
function greet() { console.log(`Hello, ${this.name}`); }
const person = { name: ‘Alice’ };
greet.call(person);   // 输出 Hello, Alice
greet.apply(person);  // 同上，传参方式不同
const boundGreet = greet.bind(person); // bind返回一个新函数
boundGreet();         // 输出 Hello, Alice
```
- `new`绑定：用`new`关键字调用构造函数。`this`指向新创建的那个实例对象。
```javascript
function Person(name) {
    this.name = name; // this指向新创建的对象
}
const p = new Person(‘Bob’);
console.log(p.name); // Bob
```
- 箭头函数：没有自己的`this`。它的`this`继承自定义时所在的外层普通函数的`this`，且一旦绑定就无法更改。
```javascript
const obj = {
    name: ‘obj’,
    regular: function() {
        console.log(this.name); // 指向obj
        const arrow = () => {
            console.log(this.name); // 继承自regular的this，也指向obj
        };
        arrow();
    },
    arrowFunc: () => {
        console.log(this.name); // 定义在对象字面量中，外层是全局，this指向window
    }
};
obj.regular(); // 输出 ‘obj’, ‘obj’
obj.arrowFunc(); // 输出 ‘’ (window.name)
```

所以整体规则就是：先看函数怎么被调用。1. new调用 -> 新对象。2. call/apply/bind调用 -> 指定的对象。3. 对象方法调用 -> 该对象。4. 普通函数调用 -> 全局（或undefined）。箭头函数看外层。

### 原型与原型链
一样是重点，很多公司喜欢考这种底层逻辑

首先你要明确的是：
- 每个函数都有一个 `prototype`属性（原型对象）。
- 每个对象都有一个 `__proto__`属性（指向其构造函数的原型对象）。
- 用`new`调用构造函数创建的实例，其` __proto__`指向该构造函数的 `prototype`。
```javascript
function Person(name) { // 构造函数
    this.name = name;
}
// 在原型对象上添加方法，所有实例共享
Person.prototype.sayHello = function() {
    console.log(`Hello, I‘m ${this.name}`);
};

const p1 = new Person(‘小明’);
const p2 = new Person(‘小红’);

console.log(p1.__proto__ === Person.prototype); // true
console.log(Person.prototype.constructor === Person); // true
p1.sayHello(); // Hello, I‘m 小明
p2.sayHello(); // Hello, I‘m 小红
// p1和p2共享同一个sayHello方法，节省内存。
```

**原型链**

当访问一个对象的属性时，如果对象自身没有，就会通过 ``__proto__``去它的原型上找，如果原型上也没有，就继续去原型的原型上找，直到找到 Object.prototype（最顶层，其``__proto__``为null）。这条查找链路就是原型链。这就是JavaScript继承的基础。
```javascript
console.log(p1.toString()); // p1自身和Person.prototype上都没有toString
// 沿着原型链找到Object.prototype.toString，所以可以调用。
```

手写 new 操作符（经典面试题）:
```javascript
我也不会
```
反正大致思路就是:将新的对象原型链指向它的prototype，然后构造新对象将this绑定一下，最后判断,如果构造函数返回了一个对象，则返回该对象；否则返回新创建的对象
```javascript
function myNew(constructor, ...args) {
    // 1. 创建一个空对象，并将其原型指向构造函数的prototype
    const obj = Object.create(constructor.prototype);
    // 2. 执行构造函数，并将this绑定到这个新对象
    const result = constructor.apply(obj, args);
    // 3. 如果构造函数返回了一个对象，则返回该对象；否则返回新创建的对象
    return (typeof result === ‘object’ && result !== null) ? result : obj;
}
```

### 事件循环与异步编程

首先，我认为可以在脑中建立一个“运行时”模型，后面再讲

js是单线程的，但要处理网络请求、定时器、用户点击等多种异步任务，如何做到不卡顿？靠的就是事件循环。

**核心概念：调用栈、Web APIs、任务队列**
想象一下：你家里只有一个厨师（主线程）的餐厅，他一次只能做一道菜（执行一个任务）。但他很聪明，遇到需要长时间等待的事情（比如炖汤），就让助手（Web APIs/Node APIs）去盯着，自己继续做别的菜。等助手说“汤炖好了”，厨师就把炖汤的后续步骤（回调函数）记在小本子（任务队列）上，等手头的事一做完，就按顺序处理小本子上的任务。

**调用栈**：主线程执行任务的地方，是一个**栈**结构（先进后出）。每次调用函数，就将其推入栈顶，执行完毕就弹出。同步代码在这里顺序执行。

**Web APIs**:由浏览器或Node.js提供的额外能力，比如`setTimeout`, DOM事件, AJAX (fetch, XMLHttpRequest), 文件读写。它们不是JavaScript引擎的一部分。当JS代码调用 `setTimeout(callback, 1000)`时，其实是告诉浏览器：“1秒后，请把这个callback函数放进任务队列”。浏览器负责计时，JS引擎继续执行后面的代码，**这就是“异步”的本质**。

**任务队列**：一个先进先出的队列，存放着准备就绪、等待主线程执行的回调函数。它又分为：
- 宏任务队列：包含 setTimeout, setInterval, setImmediate(Node), I/O操作, UI渲染, MessageChannel等回调。
- 微任务队列：包含 Promise.then/catch/finally, MutationObserver, process.nextTick(Node, 优先级最高) 的回调。

这里你其实要着重记住：setTimeout是宏任务，Promise.then/catch/finally是微任务

**事件循环**

js的事件循环就是在不断的做以下事情：

1. **执行同步代码**：从调用栈顶部开始，执行所有同步任务，直到调用栈被清空。
2. 检查并**执行所有微任务**：调用栈清空后，事件循环会立即、依次执行当前微任务队列里的所有微任务。关键点：如果在执行一个微任务的过程中，又产生了新的微任务，这个**新微任务也会被加入到当前队列的末尾，并在本轮循环中被执行**，直到微任务队列完全清空。这就是“微任务的无限递归”风险。
3. 渲染页面（如有需要）：浏览器可能会在这个时机进行UI渲染（回流、重绘）。
4. 取出**一个宏任务**执行：从宏任务队列的队首取出一个宏任务，将其对应的回调函数推入调用栈，开始执行（回到第1步）。
5. 重复此循环。

即一次事件循环，处理一个宏任务和所有微任务。

ok，那现在可以用一个例题来解释：

```javascript
console.log(‘1’); // 同步任务

setTimeout(function callback1() {
    console.log(‘2’); // 宏任务
    Promise.resolve().then(function p1() {
        console.log(‘3’); // 微任务 (在宏任务callback1内部产生)
    });
}, 0);

Promise.resolve().then(function p2() {
    console.log(‘4’); // 微任务
});

console.log(‘5’); // 同步任务
```

分解一下：

1. 初始状态：调用栈执行全局同步代码。
2. 执行 `console.log(‘1’)`，打印 1。
3. 遇到 `setTimeout`，JS引擎将其回调函数 `callback1`交给浏览器的定时器线程，并开始计时（0ms）。然后立即返回，继续执行下一行。此时 `callback1`在Web API环境中等候。
4. 遇到 `Promise.resolve().then(p2)`。`Promise.resolve()`是同步的，立即返回一个已解决的`Promise`。`.then(p2)`将它的回调函数 `p2`推入微任务队列。
5. 执行 `console.log(‘5’)`，打印 5。
6. 此时，全局同步代码全部执行完毕，调用栈为空。事件循环开始检查微任务队列。
7. 微任务队列中有一个任务 p2。将其推入调用栈执行，打印 4。执行完毕，弹出。再次检查微任务队列，已空。
8. 第一个事件循环结束。此时，浏览器的定时器线程可能已经计时完成，它将 `callback1`推入宏任务队列。
9. 事件循环检查宏任务队列，发现 `callback1`，将其推入调用栈执行。
10. 执行 `callback1`内部代码：

    - `console.log(‘2’)`，打印 2。
    - 遇到 `Promise.resolve().then(p1)`，将回调 p1推入微任务队列。
11. `callback1`这个宏任务执行完毕，调用栈再次为空。事件循环再次进入“检查微任务”阶段。
12. 发现微任务队列有任务 p1，将其推入调用栈执行，打印 3。执行完毕，微任务队列清空。
13. 继续检查宏任务队列，已空。程序等待新的任务。

最终输出顺序：1，5，4，2，3

**async/await在事件循环的表现**

- `async`函数永远返回一个 `Promise`。函数内部的返回值会被包装成一个 resolved 的 Promise

- `await`表达式会暂停​ `async`函数的执行，等待其后的 `Promise` 状态落定（resolved 或 rejected）。关键：await后面的代码，相当于被放进了 `.then()`回调里，也就是微任务。

```javascript
async function async1() {
    console.log(‘async1 start’); // 1. 同步
    await async2(); // 2. 执行async2，并将后面的代码作为微任务
    console.log(‘async1 end’); // 5. 微任务
}
async function async2() {
    console.log(‘async2’); // 3. 同步
}
console.log(‘script start’); // 0. 同步
async1();
new Promise(resolve => {
    console.log(‘promise1’); // 4. 同步 (Promise构造函数是同步的)
    resolve();
}).then(() => {
    console.log(‘promise2’); // 6. 微任务
});
console.log(‘script end’); // 7. 同步
// 输出：script start -> async1 start -> async2 -> promise1 -> script end -> async1 end -> promise2
```

1. 先执行所有同步代码，按顺序打印：`script start`, `async1 start`, `async2`, `promise1`, `script end`。

2. 执行 `await async2()`时，它相当于 `Promise.resolve(async2()).then(...)`。`async2()`执行完（打印`async2`）后，`await`下面的 `console.log(‘async1 end’)`被放入微任务队列。

3. `Promise.then`的回调也被放入微任务队列。

4. 同步代码执行完毕，开始清空微任务队列。队列里有两个微任务：`async1 end`和 `promise2`。由于 `async1 end`先入队，所以先打印它，再打印 `promise2`。

### 模块化

模块化解决了什么？

代码组织、命名冲突、按需加载等

1. `CommonJS`：`Node.js` 的默认模块系统。同步加载，运行时加载。通过 `module.exports`导出，`require()`导入。**导出的值是值的拷贝**。
```javascript
// math.js
module.exports = { add: (a, b) => a + b };
// 或 exports.add = (a, b) => a + b;
// app.js
const math = require(‘./math.js’);
```

2. `ES6 Module (ESM)`：现代前端标准。异步加载，编译时静态分析。通过 `export`导出，`import`导入。**导出的值是值的引用（活的绑定）**。
```javascript
// math.mjs (或 .js 在支持ESM的环境中)
export const add = (a, b) => a + b;
export default function multiply(a, b) { return a * b; }
// app.mjs
import mult, { add } from ‘./math.mjs’;
```

区别：
1. CommonJS是运行时加载（动态），ESM是编译时加载（静态，可以进行tree shaking）

    - `tree shaking`：webpack中可以用于描述消除无用代码的过程。通过静态分析哪些模块和变量未被使用或引用，然后删除这些代码。

2. CommonJS 输出的是值的拷贝，模块内部变化不影响已导入的值。ESM 输出的是值的引用，原模块内部变化，导入方能看到。

3. this指向：CommonJS模块顶层this指向当前模块，ESM模块顶层this是undefined。

所以能得知，require是同步加载模块，而原生浏览器ESM`<script type=“module”>`，是异步加载的

### 错误处理与性能

这里大致点出常见的错误处理：

- `try ... catch`捕获`async`函数内部的`await`错误

- `Promise`的错误要用 `.catch()`或在 `async`函数内用 `try...catch`来捕获。

- 全局错误捕获：
    - `window.onerror`/ `window.addEventListener(‘error’, handler)`：捕获JS运行时错误、资源加载失败。
    - `window.addEventListener(‘unhandledrejection’, handler)`：捕获未处理的`Promise`拒绝。

**内存管理**

垃圾回收：从根对象（全局对象）出发，标记所有可达对象，清除从未标记的对象

常见内存泄漏：
- 意外的全局变量：如函数内未声明的变量会变成`window`属性
- 被遗忘的定时器或回调：`setInterval`忘记清除，事件监听器忘记移除
- 闭包引用：函数内部引用外部变量，导致外部变量无法释放
- DOM 引用：JS中保留了DOM元素的引用，即使从页面上移除了该元素

**防抖**

防抖就是事件触发后，等待一段时间（比如n秒），如果在这段时间内事件没有再次触发，那么就执行函数；如果在这段时间内事件又触发了，那么就重新开始计时，等待一段时间后再执行。一般用在搜索框自动搜索

```javascript
function debounce(func, wait) {
  let timeout;  // 用于保存定时器
  return function(...args) {  // 返回一个函数，这个函数就是事件处理函数
    const context = this;  // 保存this，以便后续调用func时能够绑定正确的this
    clearTimeout(timeout);  // 清除之前的定时器
    // 设置一个新的定时器，在wait秒后执行func
    timeout = setTimeout(() => {
      func.apply(context, args);  // 调用我们想要防抖的函数，并传入参数
    }, wait);
  };
}
```

使用示例：
```javascript
假设我们有一个搜索框，我们希望当用户停止输入500毫秒后才执行搜索函数。
function search(query) {
  console.log(`搜索: ${query}`);
}

const debouncedSearch = debounce(search, 500);

// 模拟输入事件
const input = document.getElementById('search-input');
input.addEventListener('input', function(event) {
  debouncedSearch(event.target.value);
});
```

1. 用户输入第一个字符，触发input事件，调用debouncedSearch，此时会清除之前的定时器（如果有的话），然后设置一个500ms后执行search的定时器。
2. 在500ms内，用户又输入了第二个字符，再次触发input事件，调用debouncedSearch，这时会清除上一个定时器（也就是第一个字符设置的定时器），然后重新设置一个500ms的定时器。
3. 如此反复，直到用户停止输入500ms后，最后一个定时器没有被清除，然后执行search函数。

**节流**

节流的核心思想：事件持续触发时，每隔一段时间（比如n秒）执行一次函数，也就是稀释函数的执行频率。一般用于滚动加载：滚动过程中每隔一段时间检查一次位置，而不是每次滚动都检查。或按钮频繁点击：防止用户快速点击按钮多次提交。

**总结**

防抖：将多次执行合并为一次执行，只在事件停止触发后执行一次。适用于只需要处理最后一次触发事件的场景。

节流：每隔一段时间执行一次，稀释执行频率。适用于需要均匀执行事件的场景。

## 第三部分 React

### 核心思想

**声明式与命令式**

- 命令式：关注“怎么做”，一步步指挥计算机。就像用积木造房子，你得手动把每块积木放到正确的位置。

- 声明式：关注“做什么”，只描述最终状态，由框架负责实现。就像画房子的设计图，React 这个“施工队”帮你把房子盖出来，下次你改了设计图，React 会聪明地只修改有变化的部分。

React 的优势：你不用再手动操作DOM，只需关心状态（数据）的变化。这让代码更可预测、更易于维护。

**JSX**

jsx不是html，是`React.createElement(component, props, ...children)`的语法糖

const element = <h1 className="greeting">Hello, world!</h1>;

// 经过Babel等工具编译后
const element = React.createElement(

// React.createElement 返回的是一个React元素（虚拟DOM）对象

**组件化**

组件是可复用的、独立的UI片段。React 有两种组件：

- 类组件：ES6 class，有生命周期和内部状态 (`this.state`)。

- 函数组件：纯函数，接收 `props`并返回JSX。在 React 16.8 (Hooks) 之后，函数组件也能拥有状态和生命周期。

### 组件、状态、Hooks

现在其实函数组件用的更多，而类组件的`props`、`state`用的越来越少，所以我就先不讲类组件的东西

Hooks的产生，你在不编写 `class` 的情况下使用 `state` 和其他 `React` 特性。核心思想是将状态逻辑与组件解耦，使其可复用。

1. `useState`：在函数组件中添加状态
```javascript
import React, { useState } from ‘react’;
function Counter() {
  // useState 返回一个数组: [当前状态值, 更新状态的函数]
  const [count, setCount] = useState(0); // 初始状态为0
  // 更新状态，触发重新渲染
  return (
    <div>
      <p>You clicked {count} times</p>
      <button onClick={() => setCount(count + 1)}>Click me</button>
    </div>
  );
}
```

2. `useEffect`：处理副作用（生命周期替代品）

- 副作用：数据获取、订阅、手动修改DOM等任何与渲染结果无关的操作。
- useEffect接收一个函数（副作用函数）和一个可选的依赖数组。
- 执行时机：在组件渲染到屏幕之后执行（异步，不阻塞浏览器绘制）。

```javascript
useEffect(() => {
  // 1. 无依赖数组：每次渲染后都执行
});

useEffect(() => {
  // 2. 空依赖数组 []：仅在组件挂载后执行一次，类似 componentDidMount
  const subscription = props.source.subscribe();
  return () => {
    // 清理函数：在组件卸载前执行，类似 componentWillUnmount
    subscription.unsubscribe();
  };
}, []);

useEffect(() => {
  // 3. 有依赖数组 [props.source]：仅在 props.source 变化时执行
  const subscription = props.source.subscribe();
  return () => {
    // 清理函数：在下次 effect 执行前，会先执行这个清理函数
    subscription.unsubscribe();
  };
}, [props.source]); // 依赖项
```

**依赖数组必须包含副作用函数中用到的、且会发生变化的所有props、state 和 context 值，否则会引用到旧值。**

3. `useContext`：跨越组件层级直接获取数据
- 解决“prop drilling”（层层传递props）的问题。

```javascript
// 1. 创建 Context
const ThemeContext = React.createContext(‘light’);
// 2. 在顶层组件提供值
function App() {
  return (
    <ThemeContext.Provider value="dark">
      <Toolbar />
    </ThemeContext.Provider>
  );
}
// 3. 在深层子组件消费值
function Toolbar() {
  const theme = useContext(ThemeContext); // 直接获取，无需层层传递
  return <div style={{ background: theme === ‘dark’ ? ‘#333’ : ‘#FFF’ }}>...</div>;
}
```

4. `useRef`：引用与持久化
- 访问DOM节点：最常见的用途。
```javascript
function TextInput() {
  const inputEl = useRef(null);
  const onButtonClick = () => {
    inputEl.current.focus(); // 直接操作DOM
  };
  return (
    <>
      <input ref={inputEl} type="text" />
      <button onClick={onButtonClick}>Focus the input</button>
    </>
  );
}
```

- 存储可变值：`.current`属性的变化不会触发重新渲染。适合存储不需要触发渲染的变量（如定时器ID、上一次的值等）。
```javascript
function Timer() {
  const intervalRef = useRef(); // 存储定时器ID
  useEffect(() => {
    intervalRef.current = setInterval(() => { /* ... */ }, 1000);
    return () => clearInterval(intervalRef.current);
  }, []);
}
```

5. `useMemo`与 `useCallback`：性能优化双刃剑

- 它们都是用来“缓存”的，目的是避免在每次渲染时都进行高开销的计算或创建新的函数，只有当依赖项变化时才重新计算/创建。
    - `useMemo`：缓存计算结果。\
    ```javascript
    const memoizedValue = useMemo(() => computeExpensiveValue(a, b), [a, b]);
    // 只有 a 或 b 变化时，才会重新执行 computeExpensiveValue
    // 否则直接返回上一次缓存的值
    ```

    - `useCallback`：缓存函数本身。
    ```javascript
    const memoizedCallback = useCallback(
    () => { doSomething(a, b); },
    [a, b],
    );
    // 等价于 useMemo(() => () => doSomething(a, b), [a, b])
    // 只有当 a 或 b 变化时，才会返回新的函数，否则返回缓存的旧函数
    ```

- 为什么需要？​ 当一个函数作为 prop 传递给子组件，并且子组件用 React.memo包裹时，如果父组件重新渲染，会创建一个新的函数，导致子组件的 props 变化，从而触发子组件不必要的重新渲染。useCallback可以保证函数引用不变，避免这种情况。

6. 自定义 Hook：逻辑复用的终极武器

- 自定义 Hook 是一个以 use开头的函数，其内部可以调用其他 Hook。它让你在不增加组件层级的情况下，复用状态逻辑。

```javascript
// 自定义一个获取窗口大小的 Hook
function useWindowSize() {
  const [size, setSize] = useState({ width: window.innerWidth, height: window.innerHeight });
  useEffect(() => {
    const handleResize = () => setSize({ width: window.innerWidth, height: window.innerHeight });
    window.addEventListener(‘resize’, handleResize);
    return () => window.removeEventListener(‘resize’, handleResize);
  }, []); // 空数组，只监听一次
  return size; // 返回状态
}
// 在任何组件中使用
function MyComponent() {
  const { width, height } = useWindowSize();
  return <div>Window size: {width} x {height}</div>;
}
```

### 虚拟DOM与Diff算法（React的高效核心）

1. 什么是虚拟DOM？

虚拟DOM是一个轻量级的JavaScript对象，是真实DOM的内存表示。当组件状态变化时，React会重新构建整个虚拟DOM树（这个开销很小，因为只是JS对象操作）。

2. Diff 算法（协调）

React 将新旧两棵虚拟DOM树进行比较，计算出最小必要的DOM更新操作，这个过程叫做“协调”或“Diffing”。其核心策略是：
- 同层比较：React 只会对同一层级的节点进行比较，不会跨层级移动。这使得算法复杂度从 O(n³) 降为 O(n)。
- 类型不同，直接替换：如果节点类型（如从 <div>变成 <span>）不同，React 会直接销毁旧节点及其子树，创建新节点。
- 类型相同，更新属性：如果节点类型相同（如都是 `<div className=“before”>和 <div className=“after”>`），React 会保留DOM节点，仅更新有变化的属性。
- 列表对比使用 Key：当渲染一组动态子元素（如 map生成的列表）时，必须给每个子元素一个稳定且唯一的 key属性。React 通过 key来识别哪些元素是新的、哪些被移动了，从而高效地复用现有DOM节点，而不是重新创建。

3. 由上点提出疑问，为什么需要 Key？

假设列表 [‘A’, ‘B’, ‘C’]变成 [‘D’, ‘A’, ‘B’, ‘C’]。
- 没有 key：React 会按索引比较，认为第一个节点从 A 变成了 D，第二个从 B 变成了 A... 导致所有节点都被更新，甚至可能产生错误的副作用（如输入框状态错乱）。
- 有 key：React 通过 key 识别出 A、B、C 是已有的节点，只是头部插入了一个新节点 D。它会复用 A、B、C 的 DOM 节点，只创建 D 的新节点并插入到前面。性能大幅提升，且状态得以保留。

### 状态管理与高级模式

1. 状态提升

当多个组件需要共享状态时，将状态提升到它们最近的公共父组件中管理，然后通过 `props` 向下传递。这是最简单的数据流。

2. Context API

解决跨越多层组件传递 `props` 的繁琐问题。适合传递“全局”数据，如主题、用户信息、语言等。`useContext`让消费变得简单。

3. 状态管理库（Redux/Mobx）

当应用状态变得非常复杂，组件间通信频繁且层级很深时，Context 可能导致不必要的渲染（因为任何 Context 值的变化，都会导致所有消费它的组件重新渲染，即使它只使用了其中一部分数据）。这时需要 Redux 等库。

4. 性能优化
    - `React.memo`：类似于 `PureComponent`，用于函数组件。它对 `props` 进行浅比较，如果 `props` 没变，就跳过渲染。

    - `useMemo`/`useCallback`：如上所述，避免不必要的计算和子组件重渲染。

    - 代码分割：使用 `React.lazy`和 `Suspense`动态加载组件，减少初始包体积。

    ```javascript
    const OtherComponent = React.lazy(() => import(‘./OtherComponent’));
    function MyComponent() {
    return (
        <React.Suspense fallback={<div>Loading...</div>}>
            <OtherComponent />
        </React.Suspense>
        );
    }
    ```

由此可引出部分面试题：

1. React 组件如何通信？

    - 父子：props，回调函数。
    - 兄弟/深层：状态提升，Context API，状态管理库（Redux），事件总线（不推荐）。

2. `useEffect`的依赖项为空数组 `[]`与不传依赖项有什么区别？

    - 空数组：`effect` 仅在**组件挂载后运行一次**（`componentDidMount`），清理函数在**卸载时运行一次**（`componentWillUnmount`）。
    - 不传依赖项：`effect` 在**每次渲染后都运行**，清理函数在**每次运行 `effect` 前和卸载时运行**。

3. `React` 的渲染流程是怎样的？

    状态/`Props`改变 -> 重新执行组件函数（或调用`render`方法） -> 生成新的虚拟DOM树 -> 与旧的虚拟DOM树进行Diff比较 -> 计算出需要更新的真实DOM操作 -> 应用这些更新到真实DOM。

4. 什么是受控组件和非受控组件？
    - 受控组件：表单元素的值由 `React` 的 `state` 控制。`value={state}+ onChange={setState}`。
    - 非受控组件：表单元素的值由 DOM 自身管理，通过 `ref`来获取。通常用于文件上传等场景。

## 第四部分 HTTP

### 请求

|请求|用途|
|------|---------------------------|
|GET|**获取资源**。幂等、安全。数据在 URL 中（查询参数），长度有限|
|PUT|**完整更新资源**。幂等。用请求体中的内容替换目标资源的全部内容。如果资源不存在，有些实现会创建。|
|PATCH|**部分更新资源**。非幂等。只发送需要修改的字段。|
|DELETE|**删除资源**。幂等。|
|HEAD|与 GET 类似，但服务器**只返回响应头**，不返回响应体。用于获取资源的元信息（如检查资源是否存在、大小、最后修改时间）。|
|OPTIONS|获取目标资源支持的**通信选项**（允许的方法等）。常用于 CORS 预检请求。|
|

- 幂等：任意多次执行所产生的副作用，与一次执行的效果相同。GET、PUT、DELETE 是幂等的。这意味着你刷新页面、重复提交表单（PUT/DELETE），结果应该是一样的。POST 不是幂等的，多次提交会创建多个资源。
- 安全：不会修改服务器资源。只有 GET 和 HEAD 是安全的。安全的方法应该是只读的。

### 状态码

过一下就ok

- 1xx (信息)：临时响应，表示请求已接收，继续处理。101 Switching Protocols(升级到 WebSocket)。
- 2xx (成功)：
    - 200 OK：标准成功响应。
    - 201 Created：请求成功并创建了新资源。通常配合 POST 或 PUT 使用，响应头 Location会包含新资源的 URL。
    - 204 No Content：服务器成功处理，但无内容返回（响应体为空）。常用于 DELETE 或不需返回数据的 PUT。
    - 206 Partial Content：客户端进行了范围请求（Range请求），服务器成功返回了部分内容。用于大文件分片下载、视频拖拽播放。
- 3xx (重定向)：
    - 301 Moved Permanently：永久重定向。浏览器和搜索引擎会更新书签和索引，以后直接访问新地址。SEO权重会传递到新URL。
    - 302 Found：临时重定向。浏览器会跳转，但搜索引擎不会更新索引。最初的语义是“本次请求请临时用另一个URI访问”，但实际被广泛用作临时跳转。
    - 304 Not Modified：缓存相关。客户端发送了带条件的请求（如 If-Modified-Since），而资源未修改，服务器返回此状态码，告诉客户端可以直接用本地缓存。不包含响应体，节省带宽。
- 4xx (客户端错误)：
    - 400 Bad Request：笼统的客户端错误，服务器无法理解请求（如参数错误、格式错误）。
    - 401 Unauthorized：未认证。需要身份验证，且客户端未提供或提供的凭据错误。响应头应包含 WWW-Authenticate告知如何认证。
    - 403 Forbidden：已认证，但权限不足。服务器理解请求但拒绝执行。
    - 404 Not Found：资源不存在。可能是URL错误，或资源已被删除。
    - 405 Method Not Allowed：请求行中指定的方法不被目标资源支持。响应头应包含 Allow字段，列出支持的方法。
- 5xx (服务器错误)：
    - 500 Internal Server Error：笼统的服务器内部错误，不知道具体原因。
    - 502 Bad Gateway：作为网关或代理的服务器，从上游服务器收到无效响应。
    - 503 Service Unavailable：服务器暂时过载或维护，无法处理请求。通常可配合 Retry-After头告知客户端何时重试。
    - 504 Gateway Timeout：网关或代理服务器，未能及时从上游服务器收到响应。
 
### HTTP 缓存机制（重点）

缓存是 Web 性能优化的重中之重。目标是尽量减少不必要的网络请求

缓存分为两类：**强制缓存**和**协商缓存**。浏览器发起请求时，会先**检查强制缓存，如果失效，再走协商缓存**。

- 强制缓存：在缓存有效期内，浏览器不会发送请求到服务器，直接从本地磁盘/内存读取资源。HTTP/1.1 用 `Cache-Control`控制。

    - `Cache-Control`响应头​ (常用指令)：
        - `max-age=3600`：资源在客户端缓存的有效期，单位秒。从请求成功的时间点开始计算。
        - `no-cache`：不是不缓存，而是禁用强制缓存。每次使用缓存前，必须向服务器验证（即走协商缓存）。
        - `no-store`：真正的不缓存。禁止浏览器和所有中间缓存存储任何响应内容。用于高度敏感数据。
        - `public`：响应可以被任何缓存（浏览器、CDN、代理服务器）缓存。
        - `private`：响应只能被用户的浏览器缓存，中间代理（如CDN）不能缓存。通常用于个性化数据。
        - `immutable`：资源永不变，缓存期内无需验证。适合带版本哈希的文件（如 main.abcd1234.js）。
        - `Expires(HTTP/1.0)`：指定一个绝对的过期时间（GMT格式）。已被 `Cache-Control`的 `max-age`取代，因为服务器和客户端时间可能不同步。

- 协商缓存 (对比缓存)：当强制缓存失效，或 `Cache-Control: no-cache`时，浏览器会携带缓存标识向服务器发起请求，询问资源是否变化。如果没变，服务器返回 304，浏览器用缓存；如果变了，返回 200和新资源。

    - `Last-Modified`/ `If-Modified-Since`：
        - 服务器在响应头中返回：`Last-Modified: Tue, 25 Mar 2026 10:00:00 GMT`。
        - 下次请求时，浏览器在请求头中带上：`If-Modified-Since: Tue, 25 Mar 2026 10:00:00 GMT`。
        - 服务器比较这个时间和资源的最后修改时间。如果未变，返回 304；如果变了，返回 200和新资源。
        - 缺点：1. 精度是秒，1秒内多次修改无法识别。2. 文件可能被修改了内容但时间没变（touch命令）。3. 某些服务器不能精确获取最后修改时间。
    - `ETag`/ `If-None-Match`(更精准)：
        - 服务器根据资源内容生成一个唯一标识符（哈希值），在响应头中返回：`ETag: "abcd1234"`。
        - 下次请求时，浏览器在请求头中带上：`If-None-Match: "abcd1234"`。
        - 服务器比较` ETag`。如果一致，返回 304；不一致，返回 200和新资源。
        - 强 `ETag`：字节级别的完全一致。"abcd1234"。
        - 弱 `ETag`：资源语义上相同即可，前面加 W/。`W/"abcd1234"`。允许如页脚广告变化之类的非实质性改变。

    ```bash
    浏览器发起资源请求
        ↓
    检查本地是否有缓存
        ↓
        无 ——————————————→ 向服务器请求，返回资源并缓存
        有
        ↓
    检查 `Cache-Control` 的 `max-age` 或 `Expires` 是否过期
        ↓
      未过期 ——————(强制缓存生效)—————→ 直接使用本地缓存，状态码 200 (from disk/memory cache)
        ↓
      已过期
        ↓
    浏览器携带缓存标识 (`If-None-Match`/`If-Modified-Since`) 向服务器发起请求
        ↓
    服务器比较标识，判断资源是否变化
        ↓
      未变化 ——————————————————→ 返回 304 Not Modified，浏览器使用本地缓存
        ↓
       变化
        ↓
    返回 200 OK 和新资源，并更新响应头中的缓存标识
    ````

### 跨域与CORS

跨域是浏览器的同源策略导致的。同源：协议、域名、端口完全相同。

CORS (跨源资源共享)：现代跨域解决方案。关键在于服务器端设置响应头。

- 简单请求：满足特定条件（方法为 `GET`/`POST`/`HEAD`，`Content-Type` 为 `application/x-www-form-urlencoded`、`multipart/form-data`或 `text/plain`），浏览器会直接发出请求，并在请求头中自动添加 `Origin`。服务器需要响应 `Access-Control-Allow-Origin: *`或具体的源。
- 预检请求：不满足简单请求的条件时（如使用了 `PUT`、`DELETE`或 `Content-Type: application/json`），浏览器会先发起一个 `OPTIONS`方法的预检请求，询问服务器是否允许实际请求。
    1. 预检请求头包含：`Origin`, `Access-Control-Request-Method`(实际请求的方法), `Access-Control-Request-Headers`(实际请求的自定义头)。
    2. 服务器响应必须包含：`Access-Control-Allow-Origin`, `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers`。如果允许，还可以包含 `Access-Control-Max-Age`(预检结果缓存时间)。
    3. 预检通过后，浏览器才会发出真正的请求。
- 附带身份凭证的请求：如果需要发送 `Cookie` 或 `HTTP` 认证信息，客户端需设置 `xhr.withCredentials = true`，服务器响应头必须包含 `Access-Control-Allow-Credentials: true`，并且 `Access-Control-Allow-Origin`不能为星号，必须是具体的源。

### 浏览器

浏览器输入https://www.baidu.com到显示页面然后关闭的完整流程？

思考一下这个大致流程：浏览器拿到了地址，肯定要先解析一下这个地址字符，然后验证，验证完了才能进行连接，然后发请求来渲染：

    1.URL解析：浏览器解析URL，提取协议、域名、路径等信息
    2.DNS查询：查询域名对应的IP地址（浏览器缓存→系统缓存→路由器缓存→ISP DNS→递归查询）
    3.TCP连接：三次握手建立TCP连接（SYN→SYN-ACK→ACK）
    4.TLS握手：HTTPS需要TLS握手建立安全连接
    5.HTTP请求：发送HTTP请求报文
    6.服务器处理：服务器处理请求并返回响应
    7.浏览器渲染：
        解析HTML构建DOM树
        解析CSS构建CSSOM树
        并为渲染树（Render Tree）
        布局（Layout/Reflow）计算元素位置
        绘制（Paint）将像素绘制到屏幕
        合成（Composite）层合并显示
    8.连接关闭：四次挥手关闭TCP连接
